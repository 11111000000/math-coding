# math-coding v0.992 — Skill for AI Agents

You are an AI coding agent working in a project with
math-coding v0.992 installed. The convention guides how
decisions are documented as packets.

## When to use this skill

Use this skill when:
- The user mentions "math-coding", "packet", "convention-bootstrap".
- The user asks to install or use math-coding in a project.
- The user wants to document a decision as a packet.

**When NOT to use:** Trivial changes (rename, format, typo) do
not need packets. Use git commit, not convention. See
`KNOWN_LIMITATIONS.md` for the full guide.

## The seven axioms

  A0 Difference        A4 Process
  A1 Care              A5 Accounting
  A2 Curry-Howard      A6 Self-Application
  A3 Material Basis

Read `core/spec/axioms.md` for the canonical statement of each.

## Packet (3 mandatory + 2 generated)

Mandatory:
  `packet.yaml`     manifest (lifecycle, applications[], creator, reviews[])
  `decision.md`     thesis, antithesis, synthesis
  `refinement.md`   state, operation, invariant, test

Auto-generated by `create`:
  `task.md`         generated from proposition + outcome
  `assumptions.yaml`  5 markers (status: agent-inferred)

## The seven-field spec

When you call `create`, you provide a YAML spec with seven
fields. **Only proposition + outcome are mandatory.** The
other five are strongly recommended (warning if missing, not
error). This reduces friction for trivial decisions.

| Field | Maps to | Required? |
|-------|---------|-----------|
| `proposition` | decision.md:thesis | **yes** (the claim) |
| `outcome` | task.md:outcome | **yes** (what becomes true) |
| `invariant` | refinement.md:invariant | recommended |
| `test` | refinement.md:test | recommended |
| `antithesis` | decision.md:antithesis | recommended |
| `synthesis` | decision.md:synthesis | recommended |
| `operation` | refinement.md:operation | recommended |

## Four lifecycle states

  `draft`      packet created, no SHA witness yet
  `applied`    packet has >=1 SHA in applications[] AND reviews[]
  `retired`    packet no longer applied (supersession or deprecation)
  `abandoned`  draft packet that will not be implemented

Transitions:
  create  → draft
  apply   → draft → applied
  retire  → draft|applied → retired
  abandon → draft → abandoned

## Your workflow

When the user asks to create a packet:

1. Read the context (issue, PR, discussion, code).
2. Form a proposition: one sentence, falsifiable.
3. Form an outcome: what becomes true after the decision.
4. (Recommended) form the other five fields.
5. Run `sh math-coding create <name> --from -` and pipe the spec.
6. Implement the operation in code.
7. Commit the code and the packet.
8. Run `sh math-coding apply <name>` (records SHA witness).
9. Run `sh math-coding review <name> --approve --note="..."` (peer-review).
10. Verify: `sh math-coding verify`.

When the proposition changes:

1. Run `sh math-coding retire <name> --reason=supersession`.
2. Run `sh math-coding create <new-name> --from -` with the new spec.
3. Edit new packet.yaml to add `supersession: math/<name>/`.

Or atomically:

1. Run `sh math-coding retire <name> --reason=supersession
   --supersede-with=<new-name> --from=<spec.yaml>`.

When the proposition is correct but no longer applied:

1. Run `sh math-coding retire <name> --reason=deprecation`.

When a draft will not be implemented:

1. Run `sh math-coding abandon <name> [--reason="..."]`.

When retired packets accumulate:

1. Run `sh math-coding archive <name> --confirm`.
   Moves packet to `math/archived/<name>/`. Excluded from verify.

## Stability marker (opt-in)

After a packet has been stable for 90+ days, mark it
stable: `sh math-coding stable <name>`. Sets `stable_since:`
in `packet.yaml`. No FSM impact, only metadata.

Stable markers signal "don't touch without reason". Other
agents and reviewers can see the packet is settled.

Use `sh math-coding stable <name> --unmark` to clear.

## Worked example 1: cache-ttl

User asks: "Add Redis cache with TTL."

```yaml
proposition: |
  Cache entries expire after 60 seconds, not at user request.
outcome: |
  Cache returns fresh entry within 60s of last refresh.
invariant: |
  Cache entries never served beyond TTL.
test: |
  Insert entry with ts = now - 61s. Read. Expect upstream fetch.
antithesis: |
  Manual invalidation may be needed.
synthesis: |
  TTL is fixed at 60s; manual invalidation is a separate endpoint.
operation: |
  On read, check entry timestamp. If age > 60s, refresh from upstream.
```

Then:
```
sh math-coding create cache-ttl --from /tmp/spec.yaml
git add . && git commit -m "cache: 60s TTL"
sh math-coding apply cache-ttl
sh math-coding review cache-ttl --approve --note="tests pass"
sh math-coding verify
```

## Worked example 2: antithesis categories

Read the proposition, classify it, write antithesis accordingly.

### Performance: "fast", "latency", "throughput", "cache", "TTL", "60s", "milliseconds", "p99"

Question: "What if we traded speed for correctness?"

Example: proposition "Cache returns in <10ms" → antithesis:
"The slower uncached path may be acceptable if it gives
correct results — false-fast is worse than slow-correct."

### Simplicity: "simple", "minimal", "reduce", "remove", "drop", "fewer", "smaller"

Question: "What if simplicity hides an edge case?"

Example: proposition "Remove the validation layer" → antithesis:
"The validation layer may be hiding a real edge case that
will cost more to debug than to keep."

### Safety: "validate", "check", "verify", "auth", "permission", "safe", "secure", "audit"

Question: "What if false alarms are worse than missed events?"

Example: proposition "Reject all requests with malformed
headers" → antithesis:
"False rejections may be worse than the malformed headers
they prevent — measure both sides before deciding."

### Generic

When the proposition does not fit above:
Question: "What if the opposite claim is true? What evidence
would falsify this proposition?"

Example: proposition "Use JSON for the API" → antithesis:
"A non-JSON format (protobuf, msgpack) may have better
performance and equally good tooling — measure before
deciding."

## Field checklists

Before submitting a spec to `create`, run through these
self-checks. If you cannot answer "yes" to every question,
revise the field before submitting.

### proposition
- [ ] Is it one sentence? (no "and" connecting two claims)
- [ ] Is it falsifiable? (can be wrong)
- [ ] Is it specific? (numbers, names, paths)
- [ ] Does it contain at least one concrete detail?

### outcome
- [ ] Is it observable? (you can detect when it becomes true)
- [ ] Is it measurable? (a metric, a log line, a user-visible change)
- [ ] Is it derivable from the proposition?
- [ ] Is it time-bounded or scope-bounded?

### invariant
- [ ] Is it mathematically precise? (∀, ∃, ≥, =, ∈)
- [ ] Does it hold across all transitions?
- [ ] Is it independent of the implementation?
- [ ] Is it checkable?

### test
- [ ] Is it executable? (you can run it)
- [ ] Does it cover the proposition?
- [ ] Does it have a clear expected value?
- [ ] Is it reproducible?

### antithesis
- [ ] Is it the strongest objection, not a strawman?
- [ ] Would a thoughtful reviewer raise this exact concern?
- [ ] Does it force reconsideration of the proposition?
- [ ] Is it specific to this proposition?

### synthesis
- [ ] Does it acknowledge BOTH thesis and antithesis?
- [ ] Does it explain HOW, not WHAT?
- [ ] Does it leave a path for the antithesis?
- [ ] Is it one to three sentences?

### operation
- [ ] Does it describe behavior, not implementation?
- [ ] Would it still be valid if the implementation language changed?
- [ ] Can you map it to specific code paths?
- [ ] Does it cover edge cases?

## Anti-patterns

### Proposition anti-patterns

**Vague:** "Improve performance"
**Specific:** "Reduce p99 latency by 30% on /search endpoint"

**Direction without number:** "Faster", "More secure"
**With measurable target:** "<100ms p99", "zero auth bypasses"

### Antithesis anti-patterns

**Strawman:** "What if we want it different?"
**Generic:** "This might not work in all cases."
**Tautological:** "What if the opposite is true?"
**Strong:** "Manual invalidation endpoint may be needed;
60s TTL forces cache-busting operations to wait, which is
unacceptable for security-sensitive endpoints."

### Synthesis anti-patterns

**Restates choice:** "We chose TTL."
**Ignores antithesis:** "We chose TTL because it's better."
**Acknowledges both:** "TTL is fixed at 60s; manual
invalidation is a separate endpoint. Both paths are
independent."

### Operation anti-patterns

**Implementation detail:** "Use a dict with timestamps."
**Stack-specific:** "Use Redis with EXPIRE 60."
**Behavior:** "On read, check timestamp. If age > 60s,
refresh from upstream and update timestamp."

### Test anti-patterns

**Vague:** "Test that it works."
**Always-true:** "Assert True"
**Specific:** "Insert entry with ts = now - 61s. Read.
Assert: upstream fetch happened and entry.ts updated."

### Epistemic marker anti-patterns

**fact without evidence:** "status: fact\n  statement: 60s
is acceptable\n  evidence: "  (empty)
**hypothesis masquerading as fact:** "status: fact\n  statement:
user likes this"  (with no measurement)
**unknown disguised as hypothesis:** "status: hypothesis\n
statement: nothing" (when you genuinely don't know)

## Peer-critique mode (optional but recommended)

After writing a spec, before submitting to `create`, you can
critique your own spec as a reviewer would. This is **same
agent, different prompt**.

Invoke yourself with peer-critique intent:

```
You are now a strict peer-reviewer. Apply the Field
Checklists to the spec you just wrote:

  [Field]: [Pass/Fail] [Reason]

For each FAIL: suggest a specific revision.

If more than one field fails: return --request-changes.
If all pass: --approve.
```

If you find issues, revise and re-critique. Repeat until
all fields pass. Then submit via `create`.

## Common mistakes

1. **Vague proposition** — must be falsifiable and specific.
2. **Weak antithesis** — must be strongest objection, not strawman.
3. **Synthesis restates choice** — must explain HOW, not WHAT.
4. **Operation describes implementation** — must describe behavior.
5. **Forgetting `apply`** — packet stays draft forever.
6. **Forgetting `review`** — applied without approval fails verify.
7. **Too many drafts** — use `abandon` for ones you won't implement.
8. **Packet for trivial change** — see `KNOWN_LIMITATIONS.md`.
